1. Overview

The Scoring Calibration Audit is an analysis module within TalkScore Hub that evaluates how well the AI scoring system is differentiating candidates. It detects score compression — when scores cluster in a narrow range, making it impossible to distinguish strong candidates from weak ones.

This is the central question: Is TalkScore producing useful scores, or is everyone getting a 3.5?

Why This Matters

  • Score compression is the #1 silent failure mode in AI scoring. Everything looks fine until a hiring manager realizes they can't rank 50 candidates because they all scored between 6.8 and 7.2
  • A standard deviation below the target threshold means the scoring system has lost its ability to differentiate — scores are decorative, not predictive
  • Root causes are usually fixable: vague rubrics, behavioral (not situational) questions, or insufficient probing

Scope

DimensionDescription
Date range User-selectable: last 7 days, 14 days, 30 days, custom range
Filter: Assessment Single assessment, or all assessments under the tenant
Filter: Skill Overall assessment score, or drill into a specific skill
Minimum sample size Requires at least 20 analyzed calls to generate a calibration report

2. What It Measures

The calibration audit evaluates scoring health across multiple statistical dimensions:

Core Metrics

MetricWhat It Tells YouHealthy TargetRed Flag
Standard Deviation How spread out scores are around the mean > 1.5 (on 0-10 scale) < 0.8 — scores are compressed
Score Range Used What % of the 0-10 scale is actually being used > 60% of scale (6+ point spread) < 30% — AI is only using a narrow band
Differentiation Power Can we separate top 20% from bottom 20%? Clear separation (> 2 points between top/bottom quintiles) < 1 point gap — can't distinguish candidates
Distribution Shape Is it bell-curve or lopsided? Roughly normal, using full scale Extreme clustering around one value
Kurtosis How peaked/flat the distribution is Near 3 (normal) > 6 — hyper-concentrated spike

Per-Skill Breakdown

Each metric is computed both for the overall assessment score and per configured skill. This reveals which skills differentiate well (e.g., "Technical Knowledge" with SD = 1.8) vs. which are flat (e.g., "Empathy" with SD = 0.3).

Healthy scoring vs. compressed scoring: In a healthy system, a pool of 100 candidates will have scores genuinely spread across the range — some clearly strong, some clearly weak, most in between. In a compressed system, 89% of candidates score within 1 point of each other. The rubric is broken, not the candidates.

Expected Standard Deviation by Scale

The target SD depends on the scoring scale configured for the assessment:

1 - 5 (IELTS-style)4 points effective> 0.7< 0.4

4. Computation Engine

Calibration reports can be generated on-demand or scheduled. The computation runs server-side and stores results in calibration_reports.

Step 1: Gather Scores

// src/services/calibration.ts

export async function gatherScores(
  companyId: string,
  dateFrom: Date,
  dateTo: Date,
  assessmentId?: string
): Promise<CalibrationInput> {
  let query = supabaseAdmin
    .from('analyses')
    .select(`
      talkscore_overall,
      skill_evaluations,
      skill_average_score,
      calls!inner (assessment_id, call_date)
    `)
    .eq('company_id', companyId)
    .gte('calls.call_date', dateFrom.toISOString())
    .lte('calls.call_date', dateTo.toISOString());

  if (assessmentId) {
    query = query.eq('calls.assessment_id', assessmentId);
  }

  const { data } = await query;
  return data;
}

Step 2: Compute Statistics

// Pure math — no external dependencies

export function computeCalibrationStats(scores: number[], maxScore: number = 10) {
  const n = scores.length;
  if (n < 20) return null;  // minimum sample size

  const mean = scores.reduce((a, b) => a + b, 0) / n;

  // Standard deviation
  const variance = scores.reduce((sum, s) => sum + (s - mean) ** 2, 0) / (n - 1);
  const stdDev = Math.sqrt(variance);

  // Range
  const min = Math.min(...scores);
  const max = Math.max(...scores);
  const rangePct = ((max - min) / maxScore) * 100;

  // Kurtosis (excess kurtosis — normal = 0)
  const m4 = scores.reduce((sum, s) => sum + (s - mean) ** 4, 0) / n;
  const kurtosis = (m4 / variance ** 2) - 3;

  // Skewness
  const m3 = scores.reduce((sum, s) => sum + (s - mean) ** 3, 0) / n;
  const skewness = m3 / (stdDev ** 3);

  // Differentiation gap: top 20% avg minus bottom 20% avg
  const sorted = [...scores].sort((a, b) => a - b);
  const quintile = Math.ceil(n * 0.2);
  const bottom20 = sorted.slice(0, quintile);
  const top20 = sorted.slice(-quintile);
  const diffGap = avg(top20) - avg(bottom20);

  // Health status based on target SD for this scale
  const targetSD = maxScore <= 5 ? 0.9 : 1.5;
  const compressionThreshold = maxScore <= 5 ? 0.5 : 0.8;
  const healthStatus =
    stdDev >= targetSD ? 'healthy' :
    stdDev >= compressionThreshold ? 'warning' :
    'compressed';

  // Histogram bins
  const bins = Array.from({ length: maxScore + 1 }, (_, i) => ({
    bin: `${i}-${i + 1}`,
    count: scores.filter(s => s >= i && s < i + 1).length,
  }));

  return {
    mean, stdDev, min, max, rangePct, kurtosis, skewness,
    diffGap, healthStatus, targetSD, bins,
    varianceRatio: +(stdDev / targetSD).toFixed(2),
  };
}

Step 3: Run Per-Skill Analysis

export function computeSkillCalibrations(
  analyses: AnalysisRow[],
  assessmentSkills: AssessmentSkill[]
): SkillCalibration[] {
  return assessmentSkills.map(skill => {
    // Extract this skill's scores from all analyses
    const skillScores = analyses
      .map(a => a.skill_evaluations?.find(se => se.skill_id === skill.skill_id))
      .filter(Boolean)
      .map(se => se.score);

    if (skillScores.length < 20) return { skill_id: skill.skill_id, insufficient_data: true };

    const stats = computeCalibrationStats(skillScores, skill.max_score);

    return {
      skill_id: skill.skill_id,
      name: skill.name,
      max_score: skill.max_score,
      ...stats,
      sample_scores: skillScores.slice(-10),  // last 10 for visual dots
    };
  });
}

Step 4: AI Root Cause Analysis (when compressed)

When any metric crosses the compression threshold, the system triggers an AI analysis to identify why scores are flat and how to fix it. See Section 7 for the full prompt.

Performance note: Stats computation is pure math and runs in <50ms even for 3,000 records. The AI root cause analysis is the expensive step (~3-5s, ~2000 tokens) and only runs when compression is detected.

7. AI Root Cause Analysis

When the calibration engine detects warning or compressed status, it triggers an AI analysis to diagnose the root cause and generate actionable fix recommendations. This runs via the same OpenRouter infrastructure used for call analysis.

When It Triggers

  • overall_std_dev < target threshold
  • Any skill_calibration has health_status: "compressed"
  • differentiation_gap < 1.0 (on 0-10 scale)

What Gets Sent to the AI

The AI receives the full calibration context (not transcripts — just statistics):

const CALIBRATION_ANALYSIS_PROMPT = `You are a scoring calibration expert
for AI-powered recruitment assessments. You analyze statistical patterns in
candidate scores to identify WHY scores are compressed and HOW to fix it.

You will receive:
1. Overall scoring statistics (mean, SD, range, kurtosis, distribution)
2. Per-skill breakdown with their individual statistics
3. The assessment's skill definitions and scoring criteria
4. Sample scores from recent candidates

Your job is to:
A) Identify ROOT CAUSES for score compression — be specific and actionable
B) Recommend FIXES that will improve score variance

Common root causes:
- Vague rubric criteria without behavioral anchors (what does a 2 vs 8 actually look like?)
- Behavioral questions (past experiences) instead of situational (solve-it-now)
- No follow-up probing to test depth
- AI defaulting to center when uncertain (symptom of vague criteria)
- Scoring criteria that measure self-reported traits (e.g., "shows empathy")
  instead of demonstrated behavior
- Criteria that are too similar (overlap means same signal scored twice)
- Scale too fine (0-10 when 1-5 would suffice) or too coarse

For each FIX, provide:
- A "before" example (current problematic rubric/question)
- An "after" example (improved version)
- Expected impact on variance

Respond in JSON only.`;

Cost efficiency: The AI root cause analysis only runs when compression is detected (<5% of reports for well-calibrated assessments). For compressed ones, it runs once per report generation, not per call. Typical cost: ~$0.02 per report (2K tokens on Claude Sonnet).

Root Cause Categories

CategoryDescriptionExample Fix
rubric Scoring criteria are vague, overlapping, or not anchored to observable behavior Add behavioral anchors: "5 = Named specific CRM tool, described exact escalation path"
question_design Questions elicit self-reported answers instead of demonstrable skills Switch from "Tell me about a time..." to situational scenarios
ai_behavior AI defaults to center scores when criteria are ambiguous Tighter criteria give the AI a clearer "ruler" — it stops hedging
scale_design Scale is too granular (0-10 for a yes/no trait) or too coarse Match scale granularity to actual observable differentiation levels

Summary

This document captures every requirement for TalkScore Hub — a real-time metrics and reporting dashboard for TalkScore assessment calls. The system receives call data via webhook from Talkpush, enriches it through the Talkpush API, runs dual parallel AI analyses via OpenRouter, stores results in Supabase, and presents them in a multi-tenant dashboard integrated with PAM (Product Access Manager).

1. Infrastructure & Environment Setup

1.1 Hosting & Runtime

INF-01 Replit Core (paid, always-on) — The application must run on Replit Core to ensure 24/7 availability. No sleep/hibernation cycles.
INF-02 Node.js + Express — Backend runtime must be Node.js with Express as the HTTP framework, consistent with the Talkpush tech stack.
INF-03 TypeScript — All server and client code must be written in TypeScript for type safety.

1.2 External Services

INF-04 Supabase Project — Create a Supabase project for PostgreSQL, Auth, RLS, and Realtime subscriptions.
INF-05 Upstash Redis — Provision an Upstash Redis instance for BullMQ job queue persistence. Requires TLS and maxRetriesPerRequest: null.
INF-06 OpenRouter Account — Active OpenRouter account with API key for AI analysis. Must support model switching between Claude Sonnet and GPT-4o fallback.

2. Database Schema & Supabase Setup

2.1 Tables

Six tables must be created in Supabase PostgreSQL:

2.1.1 companies (PAM Standard)

DB-01 Fields: id (UUID PK), name (TEXT NOT NULL), tenant_id (TEXT UNIQUE), status (TEXT, default 'active'), api_key (TEXT — Talkpush API key for enrichment), settings (JSONB, default '{}'), created_at, updated_at.
DB-02 Index on tenant_id (partial, WHERE tenant_id IS NOT NULL) for webhook company resolution.

2.1.2 profiles (PAM Standard)

DB-03 Fields: id (UUID PK, REFERENCES auth.users ON DELETE CASCADE), email (TEXT UNIQUE NOT NULL), full_name (TEXT NOT NULL), auth_source (TEXT, default 'direct' — values: 'direct', 'google_sso', 'crm_bridge'), created_at, updated_at.

2.1.3 user_roles (PAM Standard)

DB-04 Fields: id (UUID PK), user_id (UUID FK → profiles), client_id (UUID FK → companies ON DELETE CASCADE), role (TEXT, default 'user' — values: 'admin', 'manager', 'user'), created_at. UNIQUE constraint on (user_id, client_id).

2.1.4 assessments (Skill Configuration per Campaign)

DB-05 Fields: id (UUID PK), company_id (UUID FK → companies NOT NULL), talkpush_campaign_id (TEXT NOT NULL), name (TEXT NOT NULL), description (TEXT), skills (JSONB NOT NULL, default '[]'), is_active (BOOLEAN default TRUE), created_at, updated_at.
DB-06 Unique index on (company_id, talkpush_campaign_id) to prevent duplicate assessment configs per campaign per tenant.
DB-07 Skills JSONB array structure: each entry must have skill_id (TEXT), name (TEXT), description (TEXT), scoring_criteria (TEXT), max_score (INTEGER).

2.1.5 calls (TalkScore-Specific)

DB-08 Webhook fields: id (UUID PK), company_id (UUID FK NOT NULL), talkpush_candidate_id (TEXT NOT NULL), talkpush_application_id (TEXT NOT NULL), transcript (TEXT), recording_url (TEXT).
DB-09 Enrichment fields: candidate_name, candidate_phone, campaign_name, talkpush_campaign_id (TEXT), assessment_id (UUID FK → assessments), agent_name, call_duration_seconds (INTEGER), call_date (TIMESTAMPTZ).
DB-10 Processing fields: status (TEXT, default 'raw' — values: 'raw', 'enriched', 'analyzed', 'error'), error_message (TEXT), raw_webhook_payload (JSONB), enrichment_data (JSONB), created_at, updated_at.
DB-11 Indexes: (company_id, call_date DESC), (status), (talkpush_application_id).

2.1.6 analyses (TalkScore-Specific)

DB-12 Core fields: id (UUID PK), call_id (UUID FK → calls ON DELETE CASCADE), company_id (UUID FK → companies NOT NULL).
DB-13 TalkScore fields: talkscore_overall (NUMERIC 3,1 — 0.0-10.0), talkscore_breakdown (JSONB — pronunciation, fluency, grammar, vocabulary, coherence), communication_issues (JSONB), candidate_issues (JSONB), agent_issues (JSONB).
DB-14 Sentiment fields: sentiment_overall (TEXT — positive/neutral/negative), sentiment_score (NUMERIC 4,2 — -1.00 to 1.00), sentiment_breakdown (JSONB — per-segment).
DB-15 Call Stage/Dropoff fields: last_stage_reached (TEXT), total_questions_asked (INTEGER), total_questions_expected (INTEGER), call_completed (BOOLEAN), dropout_reason (TEXT — candidate_hangup, silence_timeout, error, completed, unknown), stages (JSONB — array of stage objects with reached, timestamp, duration_seconds).
DB-16 Candidate Questions fields: candidate_questions (JSONB — array of {question, timestamp, context, stage}), candidate_question_count (INTEGER, default 0).
DB-17 Skill Evaluation fields: skill_evaluations (JSONB — array of {skill_id, name, score, max_score, description, reasoning}), skill_summary (TEXT), skill_average_score (NUMERIC 4,2 — normalized 0-10 average).
DB-18 Agent Quality fields: quality_flags (JSONB — array of {flag_id, category, timestamp, evidence, explanation}), quality_flag_count (INTEGER, default 0), quality_summary (TEXT), has_critical_flags (BOOLEAN, default FALSE).
DB-19 AI metadata fields: talkscore_model, talkscore_tokens, talkscore_duration_ms, quality_model, quality_tokens, quality_duration_ms, raw_talkscore_response (JSONB), raw_quality_response (JSONB), created_at.
DB-20 Indexes: (company_id, created_at DESC), (company_id, talkscore_overall), (company_id, has_critical_flags) WHERE TRUE, (company_id, call_completed, last_stage_reached).

2.1.7 calibration_reports (Scoring Analysis)

DB-21 Fields: id (UUID PK), company_id (UUID FK NOT NULL), assessment_id (UUID FK, nullable — null = all assessments), date_from (TIMESTAMPTZ NOT NULL), date_to (TIMESTAMPTZ NOT NULL), sample_size (INTEGER NOT NULL).
DB-22 Stats fields: overall_mean, overall_std_dev, overall_min, overall_max, overall_range_pct, overall_kurtosis (NUMERIC 6,3), overall_skewness (NUMERIC 6,3), differentiation_gap (all NUMERIC 4,2).
DB-23 health_status (TEXT NOT NULL — 'healthy', 'warning', 'compressed').
DB-24 Distribution fields: distribution_bins (JSONB), target_distribution (JSONB).
DB-25 Analysis fields: skill_calibrations (JSONB — per-skill stats), root_causes (JSONB — AI-generated), recommendations (JSONB — AI-generated), ai_model_used (TEXT), ai_tokens (INTEGER), created_at.
DB-26 Indexes: (company_id, created_at DESC), (assessment_id, created_at DESC).

2.1.8 score_feedback (User Feedback on AI Scores)

DB-30 Fields: id (UUID PK), analysis_id (UUID FK → analyses NOT NULL), company_id (UUID FK → companies NOT NULL), user_id (UUID FK → profiles NOT NULL), feedback_type (TEXT — 'score_override', 'comment', 'answer_comment'), target_field (TEXT NOT NULL), original_value (JSONB), corrected_value (JSONB), comment (TEXT), status (TEXT, default 'pending' — 'pending', 'accepted', 'rejected', 'noted'), created_at, updated_at.

DB-34 RLS on both score_feedback and feedback_summaries: same company_id-based policy as other tables.

2.2 Supabase Configuration

DB-27 Enable Realtime on calls and analyses tables via Supabase Dashboard → Database → Replication.
DB-28 Enable Row Level Security on all 8 tables: companies, profiles, user_roles, assessments, calls, analyses, calibration_reports, score_feedback, feedback_summaries.
DB-29 Supabase Auth must be enabled and configured to support both email/password (direct) and admin-created users (CRM Bridge).

3. Webhook Processing

3.1 Endpoint

WH-01 POST /api/webhooks/call-complete — Single endpoint that receives call completion events from Talkpush CRM.

3.2 Authentication

WH-02 Verify X-Webhook-Secret header against WEBHOOK_SECRET environment variable. Return 401 if invalid.

3.3 Expected Payload

WH-03 Accept JSON body with fields: company (string, maps to companies.tenant_id), candidate_id (string), application_id (string), transcript (string), recording_url (string).
WH-04 Validate required fields (company, candidate_id, application_id, transcript). Return 400 with specific error if any missing.

3.4 Handler Logic (Sequential)

WH-05 Step 1 — Resolve company: Look up companies table by tenant_id = payload.company. Return 422 with {"error": "Unknown company: {name}"} if no match.
WH-06 Step 2 — Store raw call: Insert into calls table with status = 'raw' and full payload stored in raw_webhook_payload JSONB field.
WH-07 Step 3 — Enqueue enrichment job: Add job to BullMQ enrichment queue with {callId, companyId}. Configure with 3 attempts, exponential backoff starting at 5 seconds.
WH-08 Step 4 — Respond 200: Return {"received": true, "call_id": "{uuid}"}. Total handler time target: <500ms.

3.5 Performance

WH-09 The webhook handler must NEVER wait for enrichment or AI analysis to complete. It must respond immediately after storing the raw call and enqueuing the job.
WH-10 Must handle ~2 calls/minute (125/hr, ~3,000/day) without queuing or timing out.

4. Data Enrichment Pipeline

4.1 Talkpush API Calls

EN-01 Fetch candidate details: GET /candidates/{candidate_id} → extract name, phone.
EN-02 Fetch application details: GET /applications/{application_id} → extract campaign_name, campaign_id, agent_name, call_duration, call_date.
EN-03 Both API calls must run in parallel via Promise.all.
EN-04 Use per-company api_key from companies table for Talkpush API authentication (not a global key).

4.2 Assessment Resolution

EN-05 After enrichment, resolve assessment_id by looking up assessments table where company_id and talkpush_campaign_id match, and is_active = true.
EN-06 If an assessment match is found, update calls.assessment_id with the resolved UUID.
EN-07 If no assessment is configured for the campaign, assessment_id remains null — skill evaluations will be skipped during AI analysis.

4.3 Data Storage

EN-08 Update the calls record with all enriched fields: candidate_name, candidate_phone, campaign_name, talkpush_campaign_id, agent_name, call_duration_seconds, call_date.
EN-09 Store full enrichment API responses in enrichment_data JSONB for debugging.
EN-10 Update call status from 'raw' to 'enriched'.

4.4 Chaining

EN-11 After successful enrichment, enqueue an AI analysis job to the analysis queue with {callId, companyId, assessmentId}. Configure with 3 attempts, exponential backoff starting at 10 seconds.

5. AI Analysis — TalkScore + Sentiment + Stages + Candidate Questions

The first of two parallel AI analyses. A single prompt extracts four dimensions from the transcript.

5.1 OpenRouter Configuration

AI-01 Base URL: https://openrouter.ai/api/v1
AI-02 Default model: anthropic/claude-sonnet-4-20250514 (Claude Sonnet)
AI-03 Fallback model: openai/gpt-4o (used on 3rd retry attempt)
AI-04 maxTokens: 3000 (increased from 2000 to accommodate skill evaluations)
AI-05 temperature: 0.1 for consistent, reproducible scoring

5.2 Dimension 1 — TalkScore (Language Proficiency)

AI-06 Score candidates on overall communication quality: talkscore_overall (0.0–10.0, one decimal).
AI-07 Produce per-dimension breakdown scores: pronunciation, fluency, grammar, vocabulary, coherence (each 0.0–10.0).
AI-08 The overall score must be holistic, not a simple average of sub-dimensions. Use IELTS-style band calibration: 9.0-10.0 Expert, 7.0-8.9 Very Good, 5.0-6.9 Competent, 3.0-4.9 Limited, 0.0-2.9 Extremely Limited.
AI-09 Detect and list candidate issues (from candidate side): type (fluency/pronunciation/grammar/vocabulary/coherence), severity (low/medium/high), detail, and quoted example from transcript.
AI-10 Detect and list agent issues (from agent/AI side): type (clarity/tone/listening/flow/handling), severity, detail, and example.
AI-11 Detect and list communication issues (bilateral breakdowns): type (misunderstanding/overlap/silence/confusion/technical), severity, detail, and example.

5.3 Dimension 2 — Sentiment Analysis

AI-12 Produce overall sentiment: sentiment_overall (positive/neutral/negative) and sentiment_score (-1.00 to 1.00).
AI-13 Produce per-segment sentiment breakdown: array of {speaker, segment description, sentiment, score} for key conversation moments.
AI-14 Sentiment ranges: Positive (0.25 to 1.00), Neutral (-0.24 to 0.24), Negative (-1.00 to -0.25).

5.4 Dimension 3 — Call Stage / Dropoff Analysis

AI-15 Map the interview into stages: introduction, question_1 through question_N, closing, finished.
AI-16 For each stage, output: stage name, reached (boolean), timestamp (MM:SS), duration_seconds.
AI-17 Determine last_stage_reached, total_questions_asked, total_questions_expected (inferred from context, or null), call_completed (true only if reached "finished").
AI-18 Identify dropout_reason if call didn't complete: candidate_hangup, silence_timeout, error, completed, or unknown.

5.5 Dimension 4 — Candidate Questions

AI-19 Extract every question the candidate asks the agent (not the agent's interview questions).
AI-20 For each candidate question, capture: exact question text, timestamp (MM:SS), context (what prompted it), and interview stage where it occurred.
AI-21 Store as candidate_questions JSONB array, and set candidate_question_count.

5.6 Response Format

AI-22 The AI must respond in valid JSON only. No markdown, no explanation outside JSON.
AI-23 The AI response must include a 2-3 sentence summary field covering call quality and key findings.

6. AI Analysis — Agent Quality Flags

The second of two parallel AI analyses. Detects quality issues in the AI interviewer agent's behavior.

AQ-01 Detect 33 flag types across three categories: agent failures, bias/fairness violations, and candidate behavior signals.
AQ-02 For each flag detected, output: flag_id, category, timestamp in transcript, evidence (exact quote), explanation.
AQ-03 Produce quality_summary: 2-3 sentence quality assessment of agent performance.
AQ-04 Set has_critical_flags = true if any critical flags are detected (e.g., agent_loop, hallucination, bias).
AQ-05 Respond in valid JSON only. No markdown.

6.1 Parallel Execution

AQ-06 TalkScore prompt (Section 5) and Agent Quality prompt MUST run as parallel Promise.all calls to OpenRouter. Wall-clock time = max(call_1, call_2), NOT sum.
AQ-07 Both analyses must complete successfully before the call status is updated from 'enriched' to 'analyzed'.
AQ-08 If one analysis fails but the other succeeds, the entire call should be retried (not partial).

7. Dynamic Skill Evaluations

7.1 Assessment Configuration

SK-01 Each assessment can be configured with a custom set of skills to evaluate (via assessments.skills JSONB).
SK-02 Skills are dynamic — different assessments for different campaigns can evaluate completely different skills.
SK-03 Each skill must have: skill_id (unique within assessment), name, description, scoring_criteria, max_score (integer).

7.2 Dynamic Prompt Injection

SK-04 buildSkillsPromptSection(skills) function must dynamically generate a prompt section listing all skills to evaluate, their descriptions, scoring criteria, and max scores.
SK-05 If no assessment is configured for a call (assessmentId = null), no skills section is appended to the prompt, and all skill fields in the analysis are null.
SK-06 The skills prompt section must request: per-skill score (0 to max_score), description of performance, and reasoning with evidence from transcript.

7.3 Skill Score Storage

SK-07 Store AI skill results in analyses.skill_evaluations JSONB: array of {skill_id, name, score, max_score, description, reasoning}.
SK-08 Compute skill_average_score server-side: normalize each skill score to 0-10 scale ((score / max_score) * 10), then average across all skills. Round to 2 decimal places.
SK-09 Store skill_summary: overall 2-3 sentence assessment of candidate's skill performance (from AI).

7.4 Assessment Management API

SK-10 GET /api/assessments — List assessments for tenant (RLS-filtered).
SK-11 GET /api/assessments/:id — Assessment detail with full skills config.
SK-12 POST /api/assessments — Create assessment (admin/manager only). Required: talkpush_campaign_id, name, skills array.
SK-13 PUT /api/assessments/:id — Update assessment skills config (admin/manager only).
SK-14 GET /api/metrics/skills — Skill score averages across calls, filterable by assessment.

AU-01 Register TalkScore Hub as a product in PAM with slug: talkscore-hub.

AU-11 Upsert user_roles record with the role from PAM (admin/manager/user) for the resolved company.

9.3 PAM Sync Endpoints (Pull)

AU-13 GET /api/access-manager/clients — Returns all companies with name, tenant_id, status. Authenticated via ACCESS_MANAGER_API_KEY Bearer token.
AU-14 GET /api/access-manager/users — Returns users with email, full_name, role, tenant_id. MUST exclude auth_source = 'crm_bridge' users to prevent sync loops.

9.4 PAM Push Sync

AU-15 When a user is created/updated/deleted in TalkScore Hub's admin panel, push changes to PAM via POST /api/v1/users/register (or DELETE).
AU-16 PAM sync is fire-and-forget: never block TalkScore Hub operations. If PAM is down, log the error and continue. PAM can catch up via pull endpoints.
AU-17 Users with auth_source = 'crm_bridge' must NEVER be synced back to PAM (prevents infinite loop).

10. Multi-Tenancy & Row Level Security

MT-01 All data access must be tenant-isolated. A user can only see data for companies they have a role in (via user_roles junction table).
MT-02 RLS policies on calls, analyses, companies, assessments, calibration_reports: USING (company_id IN (SELECT client_id FROM user_roles WHERE user_id = auth.uid())).
MT-03 Supabase Realtime must respect RLS — each tenant only receives events for their own company's data.
MT-04 The webhook handler and background workers use supabaseAdmin (service role key, bypasses RLS) since they process data for all tenants.
MT-05 Dashboard API endpoints use Supabase JWT (respects RLS) — no additional tenant filtering needed in code.
MT-06 Foreign keys use company_id (UUID), not tenant_id (string). The tenant_id is only used for PAM sync and webhook company matching.

12. Scoring Calibration Audit

12.1 Purpose

SC-01 Detect score compression — when AI scores cluster in a narrow range, making it impossible to distinguish strong from weak candidates.

12.2 Scope & Filters

SC-02 Date range filter: last 7 days, 14 days, 30 days, or custom range.
SC-03 Assessment filter: specific assessment or all assessments under the tenant.
SC-04 Skill filter: overall assessment score or drill into a specific skill.
SC-05 Minimum sample size: require at least 20 analyzed calls to generate a calibration report. Show clear message if insufficient data.

12.3 Core Metrics

MetricHealthy TargetRed Flag
Standard Deviation — How spread out scores are> 1.5 (on 0-10 scale)< 0.8 — compressed
Score Range Used — % of 0-10 scale used> 60% (6+ point spread)< 30% — narrow band
Differentiation Power — Top 20% avg minus bottom 20% avg> 2 points gap< 1 point — can't distinguish
Distribution Shape — Bell curve or lopsidedRoughly normalExtreme clustering
Kurtosis — Peaked or flatNear 3 (normal, excess ~0)> 6 — hyper-concentrated
SC-06 All metrics must be computed both for talkscore_overall and for each configured skill individually.

12.4 Target SD by Scale

SC-07 Auto-adjust target SD based on the scoring scale: 0-5 scale → target >0.9, threshold <0.5. 0-10 scale → target >1.5, threshold <0.8. 1-5 IELTS-style → target >0.7, threshold <0.4.

12.5 Health Status Classification

SC-08 healthy: SD ≥ target SD. warning: SD ≥ compression threshold but < target. compressed: SD < compression threshold.

12.6 Computation Engine

SC-09 Step 1 — Gather Scores: Query analyses + calls (inner join) filtered by company_id, date range, and optional assessment_id. Extract talkscore_overall and skill_evaluations.
SC-10 Step 2 — Compute Statistics: For the overall scores array, compute: mean, standard deviation (sample variance n-1), min, max, range %, excess kurtosis, skewness, differentiation gap (top 20% avg − bottom 20% avg), health status, histogram bins (one per integer unit).
SC-11 Step 3 — Per-Skill Analysis: For each skill in the assessment, extract that skill's scores across all analyses, compute same stats as Step 2, including variance_ratio (actual_sd / target_sd), sample_scores (last 10 for visual dots).
SC-12 If a skill has <20 data points, mark it as insufficient_data: true instead of computing stats.
SC-13 Step 4 — AI Root Cause Analysis: Triggered when health is warning or compressed, or any skill is compressed, or differentiation_gap < 1.0.

12.7 AI Root Cause Analysis

SC-14 Send calibration statistics (NOT transcripts) to OpenRouter for root cause diagnosis. Include: overall stats, per-skill breakdown with scoring criteria, and sample scores.
SC-15 AI must identify root causes categorized as: rubric (vague criteria), question_design (wrong question types), ai_behavior (center-defaulting), scale_design (wrong granularity). Each cause includes: title, description, impact level (high/medium/low), affected skills.
SC-16 AI must generate fix recommendations with: title, description, before (current problematic rubric/question), after (improved version), expected_impact (variance lift estimate), priority number.
SC-17 AI root cause analysis only runs when compression is detected (<5% of reports). Cost target: ~$0.02 per report (~2K tokens on Claude Sonnet).

15. AI Feedback Analyzer (Daily)

A scheduled AI job that runs once per day, analyzes all accumulated user feedback across each tenant, identifies patterns in score disagreements, and proposes concrete improvements to assessment questions and evaluation criteria. This is the continuous improvement loop that makes the scoring system smarter over time.

15.1 Scheduling

FA-01 The feedback analyzer runs as a daily scheduled job (cron), not triggered per-call. Suggested schedule: once per day during low-traffic hours (e.g., 02:00 UTC).
FA-02 The job runs once per tenant that has new feedback since the last run. Skip tenants with no new feedback.
FA-03 Use BullMQ with a repeatable job (repeat: { cron: '0 2 * * *' }) or a lightweight cron library (e.g., node-cron).

15.2 Data Collection

FA-04 For each tenant, gather all score_feedback records from the last 30 days (rolling window), grouped by assessment.
FA-05 Compute aggregated statistics per assessment AND include the raw feedback items:
  • Total feedback count and score override count
  • Most corrected fields: which scores get overridden most frequently (e.g., "skill:customer_empathy" overridden 38 times, avg delta -1.8)
  • Per-skill disagreement rate: for each skill, the average AI score vs. average user-corrected score, and the magnitude of disagreement
  • Direction bias: is the AI consistently scoring too high or too low for each field?
  • Comment themes: extract common phrases/themes from feedback comments
  • Individual feedback items: include each score_feedback record with its id, feedback_type, target_field, original_value, new_value, comment, user_id (resolved to display name), and created_at. This allows the AI to cite specific feedback in its recommendations.
FA-06 Also fetch the assessment's current skill configuration (assessments.skills — names, descriptions, scoring criteria) to give the AI full context.

15.3 AI Analysis

FA-07 Send to OpenRouter: aggregated statistics + the individual feedback items (with IDs, user names, comments, score deltas) + current assessment configuration. The AI receives both aggregate patterns AND specific feedback records so it can cite real user feedback in its recommendations. Note: call transcripts are NOT included — only the feedback metadata and user comments.
FA-08 The AI must produce recommendations across three categories:
CategoryDescriptionExample Output
Question Improvements Changes to interview questions that would elicit better-differentiated candidate responses "Replace 'Tell me about a time you showed empathy' with 'A customer calls upset about a delayed delivery. Walk me through exactly how you'd handle the call.'"
Scoring Criteria Improvements Changes to skill rubrics that would make AI scoring more aligned with human judgment "Add behavioral anchors to Customer Empathy: 1-3 = generic response, no personalization; 4-6 = acknowledges issue, offers solution; 7-10 = acknowledges emotions, personalizes, proactively offers alternatives"
Assessment Structure Improvements Changes to the overall assessment design: add/remove/merge skills, adjust scales "Merge 'Active Listening' and 'Empathy' into one skill — user feedback shows they measure the same behavior, causing score inflation across two dimensions"
FA-09 Each recommendation must include:
  • category: question_improvement, criteria_improvement, structure_improvement
  • affected_skill: which skill this targets (or "overall" for assessment-level changes)
  • title: short action title
  • rationale: why this change is recommended, citing specific feedback items by ID and quoting user comments (e.g., "Feedback #fb-4a2c from Maria Reyes: 'The candidate just repeated company values from the website, that's not real empathy' — this was one of 38 overrides on Customer Empathy (avg delta -1.8), indicating the rubric doesn't distinguish rehearsed answers from genuine responses")
  • cited_feedback: array of score_feedback IDs that support this recommendation. Each entry includes feedback_id, user_name, comment_excerpt (first 120 chars), and score_delta. Minimum 2 cited items per recommendation when available.
  • before: current question/criteria text
  • after: proposed improved text
  • confidence: high, medium, low — based on volume and consistency of feedback
  • expected_impact: description of how this will improve scoring alignment

15.4 AI Prompt

FA-10 System prompt positions the AI as a scoring calibration and assessment design expert. It should:
  • Understand that user overrides represent ground truth (humans closer to correct than AI)
  • Identify systematic biases (AI always scores X too high/low)
  • Focus on actionable, specific changes (not vague advice)
  • Provide before/after examples for every recommendation
  • Consider that rubric changes affect ALL future calls, so recommendations must be robust
  • Always cite specific feedback items by ID in the rationale — never make a recommendation based on aggregated statistics alone without referencing the actual user comments that support it
  • Quote user feedback comments verbatim (or excerpted) so reviewers can trace every recommendation back to real user input
FA-11 Response format: valid JSON with recommendations array (each with cited_feedback array linking to specific feedback IDs), executive_summary (2-3 sentences referencing the most impactful feedback), and feedback_health (healthy/moderate/critical based on override volume and delta magnitude).

15.5 Storage & Output

FA-12 Store results in the feedback_summaries table (one record per assessment per run). Include all computed statistics and AI recommendations.
FA-13 Preserve historical summaries for trend tracking — are recommendations being applied? Is disagreement decreasing over time?
FA-14 If disagreement rate is critical (e.g., >30% of scores overridden, avg delta >2.0), flag the assessment for urgent review by surfacing an alert on the dashboard.

15.6 UI — Improvement Recommendations View

FA-15 A dedicated page or section showing the latest AI recommendations per assessment.
FA-16 Each recommendation displayed as a card with: category badge, title, rationale with cited feedback items (showing user name, quoted comment excerpt, and score delta — each linked to its entry in the Feedback Hub), before/after comparison, confidence badge, and an "Apply" button (for criteria improvements that can be auto-applied to the assessment config). Clicking a cited feedback item navigates to that feedback entry in the Feedback Hub (Section 14.5).
FA-17 "Apply" button for criteria improvements: on click, pre-fills the Skills Prompt Editor (Section 13) with the proposed changes. User reviews and confirms before saving.
FA-18 Show trend over time: a chart or summary showing how feedback volume and disagreement rate have changed across daily runs. This validates whether applied recommendations are working.

15.8 Cost & Performance

FA-19 The daily analyzer sends statistics + individual feedback items (IDs, comments, score deltas), but not call transcripts — token usage is moderate (~3K-6K tokens per assessment depending on feedback volume). At 10 assessments per tenant, cost is ~$0.30-0.60/day per tenant. Cap the number of individual feedback items sent to the AI at 100 per assessment (most recent), including all items in the statistics computation but only the top 100 in the raw items payload.
FA-20 Skip tenants/assessments with no new feedback since last run to minimize unnecessary API calls.
The continuous improvement loop: Users provide feedback on AI scores → daily analyzer identifies patterns and cites specific feedback items → AI proposes assessment improvements traceable to real user input → admin reviews cited feedback, clicks through to Feedback Hub for context → applies changes via Skills Prompt Editor → future calls score more accurately → fewer overrides needed. Every recommendation is grounded in specific, attributable user feedback — not abstract statistics.

Overview

TalkScore Hub is a real-time metrics and reporting dashboard for TalkScore assessment calls. It receives call data from Talkpush via webhook, enriches it with additional details from the Talkpush API, runs two parallel AI analyses via OpenRouter — (1) TalkScore language proficiency + sentiment, and (2) AI Agent Quality flags — and presents results in a multi-tenant dashboard.

Processing Pipeline

The pipeline is asynchronous. At ~2 calls/minute, the webhook must respond with 200 immediately and process in the background. AI analysis alone takes 3-10 seconds per call.

Talkpush CRM TalkScore Hub (Replit Core) ──────────── ────────────────────────── CRM Bridge ────────────► /auth/crm-bridge?token=xxx (SSO login) Validate token with PAM → create session Webhook ──────────────────► POST /api/webhooks/call-complete (call complete) │ ▼ 1. RECEIVE & STORE RAW Match company via webhook company field → lookup companies.tenant_id Store in calls table (status: raw) Respond 200 immediately │ ▼ 2. ENRICH (async job via BullMQ) Talkpush API ◄────────────► Fetch candidate details, campaign, agent info from Talkpush API Update call (status: enriched) │ ▼ 3. AI ANALYSIS (two parallel jobs via BullMQ) OpenRouter ◄──────────────► 3a. TalkScore + Sentiment (language proficiency) 3b. Agent Quality Flags (33 flag types) Both run in parallel on same transcript Store in analyses table Update call (status: analyzed) when BOTH complete │ ▼ 4. NOTIFY Push via Supabase Realtime to dashboard PAM (Product Access Manager) ───────────────────────────── ◄── GET /api/access-manager/clients (PAM pulls client list) ◄── GET /api/access-manager/users (PAM pulls user list) ──► POST /api/v1/users/register (TalkScore Hub pushes user changes) ──► POST /api/v1/auth/validate (TalkScore Hub validates CRM tokens)

Call Processing Status Flow

raw → enriched → analyzed
raw → error (on failure at any step, with error_message)

Why Async Matters

  • Webhook response time: Must respond 200 within seconds — can't wait for AI analysis
  • AI latency: OpenRouter calls take 3-10 seconds per transcript
  • Retry logic: BullMQ handles failed jobs with exponential backoff
  • Rate limiting: Queue controls concurrency to respect OpenRouter/Talkpush API limits
  • Observability: Each job has status tracking, timestamps, and error logging

PAM Integration (Product Access Manager)

TalkScore Hub integrates with the centralized Product Access Manager for authentication, user management, and tenant discovery. This enables SSO from the Talkpush CRM and centralized access control.

PAM Base URL: https://product-access-manager.vercel.app/api/v1

What PAM Provides

CapabilityHow
SSO from CRM CRM generates single-use token (60s TTL) → user redirected to /auth/crm-bridge?token=xxx → TalkScore Hub validates with PAM → creates local session
Centralized user registry TalkScore Hub pushes user create/update/delete to PAM via /users/register
Client discovery PAM pulls client list from TalkScore Hub via /api/access-manager/clients
User discovery PAM pulls user list from TalkScore Hub via /api/access-manager/users

SQL Schema

Companies (PAM Standard)

CREATE TABLE companies (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name        TEXT NOT NULL,
  tenant_id   TEXT UNIQUE,                      -- Talkpush platform tenant ID
  status      TEXT DEFAULT 'active',            -- 'active' | 'inactive'
  api_key     TEXT,                                -- Talkpush API key for enrichment
  settings    JSONB DEFAULT '{}',               -- tenant-specific config
  created_at  TIMESTAMPTZ DEFAULT now(),
  updated_at  TIMESTAMPTZ DEFAULT now()
);

CREATE INDEX idx_companies_tenant ON companies(tenant_id)
  WHERE tenant_id IS NOT NULL;

Profiles (PAM Standard)

CREATE TABLE profiles (
  id          UUID PRIMARY KEY REFERENCES auth.users(id) ON DELETE CASCADE,
  email       TEXT NOT NULL UNIQUE,             -- PAM identifies users by email
  full_name   TEXT NOT NULL,
  auth_source TEXT DEFAULT 'direct',          -- 'direct' | 'google_sso' | 'crm_bridge'
  created_at  TIMESTAMPTZ DEFAULT now(),
  updated_at  TIMESTAMPTZ DEFAULT now()
);

User Roles (PAM Standard)

CREATE TABLE user_roles (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id     UUID NOT NULL REFERENCES profiles(id) ON DELETE CASCADE,
  client_id   UUID REFERENCES companies(id) ON DELETE CASCADE,
  role        TEXT NOT NULL DEFAULT 'user',     -- 'admin' | 'manager' | 'user'
  created_at  TIMESTAMPTZ DEFAULT now(),

  UNIQUE(user_id, client_id)
);

Assessments (Skill Configuration per Campaign)

CREATE TABLE assessments (
  id                        UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  company_id                UUID NOT NULL REFERENCES companies(id),
  talkpush_campaign_id      TEXT NOT NULL,             -- maps to Talkpush campaign
  name                      TEXT NOT NULL,             -- e.g. "Customer Service Agent Assessment"
  description               TEXT,

  -- ═══ Skills to evaluate ═══
  skills                    JSONB NOT NULL DEFAULT '[]',  -- [{skill_id, name, description, scoring_criteria, max_score}]

  created_at                TIMESTAMPTZ DEFAULT now(),

CREATE UNIQUE INDEX idx_assessments_campaign ON assessments(company_id, talkpush_campaign_id);

Calls (TalkScore-Specific)

CREATE TABLE calls (
  id                        UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  company_id                UUID NOT NULL REFERENCES companies(id),

  -- From webhook (raw)
  talkpush_candidate_id     TEXT NOT NULL,
  talkpush_application_id   TEXT NOT NULL,
  transcript                TEXT,
  recording_url             TEXT,

  -- From API enrichment
  candidate_name            TEXT,
  candidate_phone           TEXT,
  campaign_name             TEXT,
  talkpush_campaign_id      TEXT,
  assessment_id             UUID REFERENCES assessments(id),  -- linked during enrichment
  agent_name                TEXT,
  call_duration_seconds     INTEGER,
  call_date                 TIMESTAMPTZ,

  -- Processing
  status                    TEXT DEFAULT 'raw',     -- raw → enriched → analyzed → error
  error_message             TEXT,

  updated_at                TIMESTAMPTZ DEFAULT now()

Analyses (TalkScore-Specific)

CREATE TABLE analyses (
  id                        UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  call_id                   UUID NOT NULL REFERENCES calls(id) ON DELETE CASCADE,
  company_id                UUID NOT NULL REFERENCES companies(id),

  -- ═══ TalkScore (language proficiency) ═══
  talkscore_overall         NUMERIC(3,1),              -- 0.0 - 10.0
  talkscore_breakdown       JSONB,                     -- {pronunciation, fluency, grammar, vocabulary, coherence}
  communication_issues      JSONB,                     -- [{type, severity, detail, timestamp_in_call}]
  candidate_issues          JSONB,                     -- issues from candidate side
  agent_issues              JSONB,                     -- issues from agent side

  -- ═══ Sentiment ═══
  sentiment_overall         TEXT,                      -- positive | neutral | negative
  sentiment_score           NUMERIC(4,2),              -- -1.00 to 1.00
  sentiment_breakdown       JSONB,                     -- per-segment sentiment

  -- ═══ Call Stage / Dropoff Analysis ═══
  last_stage_reached        TEXT,                      -- introduction | question_1 | question_N | closing | finished
  total_questions_asked     INTEGER,                   -- how many questions agent asked
  total_questions_expected  INTEGER,                   -- how many questions the interview should have
  call_completed            BOOLEAN,                   -- true if reached 'finished' stage
  dropout_reason            TEXT,                      -- candidate_hangup | silence_timeout | error | completed | unknown
  stages                    JSONB,                     -- [{stage, reached: true/false, timestamp, duration_seconds}]

  -- ═══ Candidate Questions ═══
  candidate_questions       JSONB,                     -- [{question, timestamp, context, stage}]
  candidate_question_count  INTEGER DEFAULT 0,        -- total questions asked by candidate

  -- ═══ Skill Evaluations (dynamic, per assessment config) ═══
  skill_evaluations         JSONB,                     -- [{skill_id, name, score, max_score, description, reasoning}]
  skill_summary             TEXT,                      -- overall skill evaluation summary
  skill_average_score       NUMERIC(4,2),              -- average across all skills (normalized 0-10)

  -- ═══ Agent Quality Flags (from AI Interview Agent Quality Analysis v4) ═══
  quality_flags             JSONB,                     -- [{flag_id, category, timestamp, evidence, explanation}]
  quality_flag_count        INTEGER DEFAULT 0,        -- total flags detected
  quality_summary           TEXT,                      -- 2-3 sentence quality assessment
  has_critical_flags        BOOLEAN DEFAULT FALSE,    -- agent_loop, hallucination, bias

  -- ═══ AI metadata (per-analysis tracking) ═══
  talkscore_model           TEXT,                      -- model used for TalkScore+Sentiment
  talkscore_tokens          INTEGER,                   -- total tokens (prompt+completion)
  talkscore_duration_ms     INTEGER,                   -- wall-clock time
  quality_model             TEXT,                      -- model used for Agent Quality
  quality_tokens            INTEGER,                   -- total tokens (prompt+completion)
  quality_duration_ms       INTEGER,                   -- wall-clock time
  raw_talkscore_response    JSONB,                     -- full TalkScore AI response
  raw_quality_response      JSONB,                     -- full Agent Quality AI response

  created_at                TIMESTAMPTZ DEFAULT now()
);

CREATE INDEX idx_analyses_company ON analyses(company_id, created_at DESC);
CREATE INDEX idx_analyses_score ON analyses(company_id, talkscore_overall);
CREATE INDEX idx_analyses_critical ON analyses(company_id, has_critical_flags)
  WHERE has_critical_flags = TRUE;
CREATE INDEX idx_analyses_dropoff ON analyses(company_id, call_completed, last_stage_reached);

Key Architecture Decisions

1. company_id (UUID) as FK, not tenant_id (string)

The calls and analyses tables reference companies.id (UUID), not the string tenant_id. The tenant_id is only used for PAM sync and webhook company matching. This keeps foreign keys clean and performant.

2. Webhook → company mapping

When a webhook arrives with a company identifier, we look up companies.tenant_id to resolve the local company_id UUID, then store the call against that UUID.

3. CRM Bridge users not synced back

Users arriving via CRM Bridge are marked auth_source = 'crm_bridge' and are NOT synced back to PAM (prevents infinite loop, per PAM integration guide).

4. RLS via user_roles junction table

Multi-tenant data isolation uses Supabase Row Level Security with the user_roles junction table. A user sees only data for companies they have a role in. This is the same pattern used by Contract Hub.

5. Webhook responds 200 immediately

At ~2 calls/minute, the webhook handler stores raw data and enqueues processing jobs. It never waits for API enrichment or AI analysis to complete before responding.

6. Separate analyses table (not inline on calls)

AI results are in a separate table because: (a) analysis can be re-run without duplicating call data, (b) the analysis schema may evolve independently, (c) supports future multi-model comparison (run the same call through different models).

7. Supabase over Neon

Chose Supabase (not Replit's native Neon PostgreSQL) because: (a) Auth + RLS + Realtime built-in, (b) consistent with other Talkpush products, (c) PAM integration patterns are already established for Supabase.

8. Upstash Redis over in-process queue

Chose Upstash Redis + BullMQ over a simpler in-memory queue because: (a) jobs survive Replit restarts, (b) built-in retry with exponential backoff, (c) rate limiting controls for OpenRouter API, (d) monitoring dashboard. Free tier handles 10K commands/day which is sufficient for 3K calls.
